# 운영 가이드

## 1. 문서 목적

이 문서는 현재 `jira-auto-agent` UI 기준으로 설정 방법, 배치 실행 절차, 작업 현황 확인 방법, 자주 발생하는 오류 대응 방법을 정리한 운영 가이드다.

대상 독자:

- 로컬에서 앱을 실행하고 직접 화면을 사용하는 운영자
- Jira, GitHub, 로컬 저장소 연결을 관리하는 개발자
- 배치 실행 결과와 clarification 상태를 추적해야 하는 리뷰어

## 2. 화면 구성

메인 화면은 4개 구역으로 나뉜다.

1. `Setup`
   - Jira 연결 정보 입력
   - Jira 공간별 저장소 연결 등록
   - 설정 불러오기, 검증, 저장
2. `Jira`
   - 백로그 조회
   - 이슈 선택
   - 이슈 상세와 최근 코멘트 확인
3. `Workflow`
   - Codex 실행 조건 입력
   - 배치 미리보기 확인
   - 저장소 상태 확인 및 배치 실행
4. `Workspace`
   - 최근 배치 상태 요약
   - 실행 단계 플로우
   - 실행별 상세 탭

## 3. Setup 사용법

### 3.1 Jira 연결 정보

입력 필드:

- `Jira Base URL`
- `Jira Email`
- `Jira API Token`
- `JQL`

권장 순서:

1. `Jira Base URL`에 `https://<your-domain>.atlassian.net` 형태의 루트 주소를 입력한다.
2. `Jira Email`에 API Token을 발급한 계정 이메일을 입력한다.
3. `Jira API Token`을 입력한다.
4. `JQL`에 백로그 조회 조건을 입력한다.
5. `필수값 검증`으로 누락 항목을 확인한다.
6. `보호된 저장`으로 설정을 저장한다.

주의 사항:

- Jira API Token은 화면에 다시 평문으로 표시되지 않는다.
- API Token 입력값을 비워 저장해도 기존 저장 토큰은 유지된다.

### 3.2 공간별 저장소 연결

이 앱은 전역 GitHub 저장소 하나를 쓰지 않고, Jira 공간 키별로 저장소를 연결한다.

입력 필드:

- 공간 키
- GitHub owner
- GitHub repo
- 기본 브랜치
- 로컬 저장소 경로
- 공간 전용 GitHub Token

등록 규칙:

- 공간 키는 Jira 이슈 키 앞부분을 기준으로 매칭한다.
  - 예: `DEMO-123`의 공간 키는 `DEMO`
- 로컬 저장소 경로는 `.git` 디렉터리가 있는 Git 루트여야 한다.
- GitHub Token은 공간별로 따로 저장된다.

등록 순서:

1. 공간 키와 저장소 정보를 입력한다.
2. 필요하면 공간 전용 GitHub Token을 입력한다.
3. `연결 추가`를 누른다.
4. 목록에 등록된 연결을 확인한다.

수정 방법:

- 저장된 연결의 `편집` 버튼으로 수정 모달을 연다.
- 모달에서 저장소 정보와 토큰 상태를 수정할 수 있다.
- 저장된 토큰이 있으면 새 값을 입력해 갱신하거나, 토큰 해제 토글로 제거할 수 있다.

### 3.3 설정 관련 버튼

- `저장값 불러오기`
  - 현재 DB에 저장된 Jira 설정과 공간별 연결을 화면에 채운다.
- `필수값 검증`
  - 누락된 입력과 공간 토큰 누락을 확인한다.
- `보호된 저장`
  - 암호화된 형태로 DB에 저장한다.
- `설정 가이드 보기`
  - 필드별 입력 가이드를 모달에서 보여준다.

## 4. Jira 구역 사용법

### 4.1 백로그 조회

`백로그 조회` 버튼은 저장된 Jira 설정을 사용해 `/api/jira/backlog`를 호출한다.

조회 결과:

- `Issue Key`
- `Summary`
- `Status`
- 선택 체크박스

### 4.2 이슈 선택

- 여러 이슈를 동시에 선택할 수 있다.
- 선택 요약 영역에는 선택 건수와 이슈 목록이 표시된다.
- `선택 해제`를 누르면 선택 상태가 모두 초기화된다.

### 4.3 이슈 상세

선택한 이슈 기준으로 아래 정보가 표시된다.

- 이슈 메타 정보
- 본문 설명
- 최근 코멘트

## 5. Workflow 구역 사용법

### 5.1 입력 항목

현재 UI에서 운영자가 직접 입력하는 주요 항목:

- `Codex Model`
- `Reasoning Effort`
- `작업 지시 상세`
- `수용 기준`
- `커밋 체크리스트`
- `Git Author Name`
- `Git Author Email`
- `로컬 테스트 없이 자동 커밋 허용`

현재 UI에 노출되지 않는 항목:

- `test_command`
  - hidden input으로만 유지된다.
  - 서버 payload 계약 호환을 위해 존재하지만, 현재 화면에서는 편집하지 않는다.

### 5.2 필수 입력 판단

실행 전에 사실상 중요한 입력은 아래와 같다.

- 선택된 Jira 이슈 1건 이상
- `작업 지시 상세`

상황에 따라 추가로 필요한 항목:

- 자동 커밋을 허용했고 로컬 Git 작성자 정보가 없으면 `Git Author Name`, `Git Author Email`
- clarification 질문이 생기면 해당 질문에 대한 답변

### 5.3 버튼 동작

- `레포 상태 확인`
  - 공간 키에 맞는 저장소 연결을 찾는다.
  - GitHub 저장소/브랜치 응답 상태를 확인한다.
  - 로컬 저장소 존재 여부, 워킹 트리 dirty 상태, Git 작성자 정보, Codex CLI 존재 여부를 확인한다.
- `배치 미리보기 갱신`
  - 선택 이슈 기준으로 작업 브랜치, 커밋 메시지, 큐 그룹, 연결 저장소를 미리 계산한다.
- `선택 이슈 배치 실행`
  - 선택된 이슈들을 배치로 등록하고 Codex 작업을 시작한다.

### 5.4 배치 미리보기

미리보기 카드에서 확인 가능한 항목:

- 이슈 키와 요약
- 직렬 처리 또는 병렬 처리 여부
- 매핑된 공간 키
- 연결 저장소
- 기본 브랜치
- 로컬 저장소 경로
- 작업 브랜치 제안값
- 커밋 메시지 제안값
- 큐 그룹

직렬 처리 기준:

- 같은 로컬 저장소 경로를 공유하는 이슈는 충돌 방지를 위해 직렬 처리된다.

### 5.5 자동 커밋 동작

현재 기본 동작은 다음과 같다.

- 자동 커밋 체크가 켜져 있으면 로컬 테스트 없이 커밋까지 진행한다.
- 테스트 명령은 현재 UI에서 입력받지 않는다.
- 따라서 커밋 전에 별도 수동 검증이 필요하면 실행 전 체크리스트에 명시하거나, 실행 후 직접 확인해야 한다.

## 6. Workspace 구역 사용법

배치를 실행하면 `Workspace` 카드가 활성화된다.

### 6.1 상단 제어

- `새로고침`
  - 최근 배치 목록과 현재 배치를 다시 불러온다.
- `진행 중으로 이동`
  - 실행 중, 추가 입력 대기, 실패 순으로 우선순위가 높은 run으로 포커스를 이동한다.
- `실패 탭만 보기`
  - 실패한 run만 필터링해서 본다.

### 6.2 진행 플로우

플로우 보드는 각 run의 현재 단계를 표시한다.

대표 상태:

- `queued`
- `running`
- `needs_input`
- `completed`
- `failed`

### 6.3 실행 상세 탭

각 run은 다음 상세 탭을 가진다.

- `실행 개요`
- `결과 요약`
- `질문과 코멘트`
- `산출물`
- `로그`

### 6.4 Clarification 처리

Codex가 작업 전에 정보가 더 필요하다고 판단하면 run 상태가 `needs_input`으로 바뀐다.

운영 절차:

1. `질문과 코멘트` 탭으로 이동한다.
2. 표시된 질문마다 답변을 입력한다.
3. 제출 버튼으로 답변을 전송한다.
4. Codex가 답변을 반영해 다시 판단한다.
5. 추가 질문이 없으면 실행 큐로 복귀한다.

동기화 참고:

- clarification 질문과 답변은 Jira 코멘트로 동기화를 시도한다.
- Jira 쓰기 실패가 있더라도 로컬 run 상태 자체는 별도로 유지된다.

## 7. 저장 구조

### 7.1 DB와 토큰

- `data/app.db`
  - Jira 설정
  - 공간별 저장소 연결
  - 암호화된 Jira API Token
  - 암호화된 공간별 GitHub Token

### 7.2 배치 스냅샷

- `data/workflow-batches`
  - 최근 배치 상태 JSON
- `data/workflow-runs`
  - 개별 run 상태 JSON

### 7.3 프로젝트 메모

- `docs/project-overview.md`
  - 공유 프로젝트 메모 개요
- `data/project-memory`
  - 런타임 기록

주의:

- `docs/project-overview.md`는 공유 메모 파일이라 일반 기능 변경용 문서처럼 직접 수정하지 않는 편이 안전하다.

## 10. 자주 보는 오류와 대응

### `jira_config_not_found`

원인:

- Jira 설정이 저장되지 않음

대응:

1. Setup에서 Jira 필드를 입력한다.
2. `보호된 저장`을 다시 실행한다.

### `github_config_not_found`

원인:

- 공간별 저장소 연결이 저장되지 않음

대응:

1. 공간 키별 연결을 추가한다.
2. `보호된 저장`을 다시 실행한다.

### `repo_mapping_*`

원인:

- 이슈 공간 키와 일치하는 저장소 연결이 없음
- 공간별 토큰이 없음
- 저장소 매핑 형식이 잘못됨

대응:

1. 이슈 키 앞부분의 공간 키를 확인한다.
2. 해당 공간 키로 저장소 연결을 등록한다.
3. 필요한 경우 공간 전용 GitHub Token을 등록한다.

### `local_repo_not_found`

원인:

- 지정된 로컬 경로가 없거나 `.git`이 없음

대응:

1. 로컬 저장소 경로가 Git 루트인지 확인한다.
2. 해당 경로에 저장소를 clone 했는지 확인한다.

### `codex_cli_not_found`

원인:

- 로컬 Codex 실행기가 없음

대응:

1. Codex CLI 설치 상태를 확인한다.
2. `레포 상태 확인` 결과에서 launcher 탐지 값을 본다.

### `git_identity_missing`

원인:

- 자동 커밋이 켜져 있는데 로컬 Git 작성자 정보가 없음

대응:

1. 화면의 `Git Author Name`, `Git Author Email`을 입력한다.
2. 또는 로컬 `git config user.name`, `git config user.email`을 설정한다.

### `needs_input`

원인:

- Codex가 작업 전에 추가 정보가 필요하다고 판단함

대응:

1. Workspace의 `질문과 코멘트` 탭으로 이동한다.
2. 질문에 답변을 입력한다.
3. 제출 후 상태 변화를 확인한다.

## 11. 문서 갱신 기준

다음 중 하나가 바뀌면 이 문서를 같이 갱신하는 것이 좋다.

- Setup 입력 필드 구조 변경
- 공간별 저장소 연결 정책 변경
- Workflow 입력 항목 변경
- Workspace 상세 탭 구조 변경
- 토큰 저장 방식 또는 환경 변수 변경
